US-588 Majeure Contradiction backend Backend ne valide pas raison_sociale pour Personne Morale ▶
Le frontend valide correctement qu'une PM doit avoir une raison_sociale (ligne 50 de create-tiers-modal.tsx). Mais le schéma Zod backend déclare ce champ comme optional(), sans jamais vérifier le type de personne. Un appel API direct (outil externe, v1 API) peut créer une Personne Morale avec seulement un nom.
// server/routes/tiers.ts — schéma de création (ligne ~130) const createTiersSchema = z.object({ type_personne: z.enum(['physique', 'morale']), nom: z.string().min(1).max(255), prenom: z.string().optional(), raison_sociale: z.string().max(255).optional(), // ← TOUJOURS optionnel ! ... }) // Manque : .superRefine() pour vérifier PM → raison_sociale obligatoire
Correction suggéréeAjouter un .superRefine() sur le schéma : si type_personne === 'morale' et !raison_sociale → erreur Zod "raison_sociale requise pour une personne morale".
US-807 Majeure Manquant backend Historique EDL locataire : statut occupation (En cours / Terminé) absent ▶
La spec US-807 montre une colonne "Statut" dans l'historique des lots d'un locataire : "En cours" si aucun EDL sortie signé ne couvre ce lot, "Terminé" si un EDL sortie signé existe. Le endpoint GET /api/tiers/:id/edl-history retourne le statut brut de l'EDL (brouillon/signe/infructueux) mais pas ce statut d'occupation dérivé.
// server/routes/tiers.ts — GET /:id/edl-history (ligne ~350) SELECT ei.id, ei.type, ei.sens, ei.statut, ei.date_signature, ... FROM edl_locataire el JOIN edl_inventaire ei ON ei.id = el.edl_id -- Retourne le sens (entree/sortie) mais PAS le statut occupation dérivé -- Spec : "En cours" = entrée signée sans sortie correspondante -- Spec : "Terminé" = sortie signée existe pour ce lot
Spec attendue (US-807)Colonne "Statut occupation" dérivée : requête qui vérifie l'existence d'un EDL sens=sortie + statut=signe sur le même lot pour ce locataire.
US-807 Majeure Manquant backend Historique EDL locataire : colonne "Propriétaire" absente ▶
Le même endpoint GET /api/tiers/:id/edl-history retourne lot et batiment mais pas le nom du propriétaire du lot (via lot_proprietaire → tiers). La spec montre explicitement une colonne "Propriétaire" dans la vue historique du locataire pour contextualiser chaque EDL.
// Manque dans la requête edl-history : (SELECT COALESCE(tp.prenom || ' ' || tp.nom, tp.raison_sociale) FROM lot_proprietaire lp JOIN tiers tp ON tp.id = lp.tiers_id WHERE lp.lot_id = l.id AND lp.est_principal = true LIMIT 1) as proprietaire_nom ← absent de la requête actuelle
US-595 Majeure Contradiction backend Changement date/heure mission ne déclenche pas la revalidation technicien ▶
La spec US-595 stipule : lorsqu'un admin modifie date_planifiee, heure_debut ou heure_fin d'une mission dont le technicien a déjà accepté (statut_invitation = 'accepte'), un workflow s'enclenche : soit "Demander revalidation" (→ statut_invitation retourne à 'en_attente'), soit "Confirmer d'office". Le PATCH /missions/:id modifie ces champs sans jamais interroger ni modifier mission_technicien.
// server/routes/missions.ts — PATCH /:id (ligne ~490) const allowedFields = ['date_planifiee', 'heure_debut', 'heure_fin', 'statut_rdv', 'commentaire'] // ... await query(`UPDATE mission SET ${fields} WHERE id = $... RETURNING *`, values) // Pas de : UPDATE mission_technicien SET statut_invitation = 'en_attente' ... // La spec exige un appel frontend/backend pour signaler la revalidation
Correction suggéréeAprès UPDATE mission, si date_planifiee/heure changent : si mission_technicien.statut_invitation = 'accepte' → le mettre à 'en_attente' automatiquement et dispatcher un webhook mission.replanifiee.
US-595 Majeure Manquant backend Aucun email envoyé lors de l'assignation d'un technicien ▶
La spec US-595 précise : "Paul reçoit une notification d'assignation". Le endpoint POST /api/missions/:id/assign et le PATCH technicien (ligne 629) créent un enregistrement mission_technicien sans aucun appel à Resend pour envoyer un email. L'infrastructure Resend est pourtant déjà configurée pour les invitations (voir server/services/email-service.ts).
// server/routes/missions.ts — ligne ~629 await query( `INSERT INTO mission_technicien (mission_id, user_id, ...) VALUES (...)` ) // Manque : await sendEmailTechnicienAssigne(technicienEmail, mission)
Spec US-595"Notification d'assignation envoyée au technicien avec le détail de la mission (date, heure, adresse du lot)."
US-809 Mineure Incohérence backend Fiche mandataire accessible côté backend sans check du type de workspace ▶
La spec US-809 précise que la vue mandataire est "visible uniquement pour les workspaces de type societe_edl". Le frontend applique correctement cette règle via showMandataire = workspace.type_workspace !== 'agence'. Mais le backend (GET /api/tiers/:id, GET /api/tiers/:id/lots) ne vérifie pas le type de workspace, autorisant n'importe quelle intégration API à accéder aux données mandataires d'un workspace de type agence ou bailleur.
Correction suggéréeAjouter dans les routes tiers (route mandataire) : vérifier que workspace.type_workspace === 'societe_edl' avant de retourner les données mandataire, ou filtrer role='mandataire' selon le type workspace.

US-823 Critique Bug backend + frontend Indisponibilités récurrentes ne s'affichent pas sur toutes les occurrences calendrier ▶
Un technicien crée une indisponibilité "Tous les lundis 9h-12h" (est_recurrent=true, recurrence_config.freq='weekly'). Ce bloc est stocké comme 1 seule ligne en base. Lorsque le calendrier charge les indisponibilités (GET /api/indisponibilites?date_from=X&date_to=Y), il retourne ce seul enregistrement. Résultat : l'indisponibilité n'apparaît que pour la semaine de date_debut, pas les semaines suivantes. Un technicien peut donc être assigné à tort sur une date où il est pourtant régulièrement indisponible.
// server/routes/indisponibilites.ts — GET list SELECT * FROM indisponibilite_technicien WHERE workspace_id = $1 AND date_debut <= $date_to AND date_fin >= $date_from -- Retourne 1 ligne même si l'indispo est récurrente chaque semaine -- Manque : expansion des occurrences selon recurrence_config
// recurrence_config stocké en JSONB : { "freq": "weekly", "byday": ["MO"], "until": "2026-12-31" } // Ces occurrences ne sont JAMAIS générées par le backend
CorrectionCréer une fonction expandRecurringIndispo(row, dateFrom, dateTo) qui génère les dates d'occurrence depuis le RRULE JSON. Le endpoint GET liste les retourne comme occurrences virtuelles. Alternative plus robuste : utiliser la lib rrule (npm) côté backend.
US-823 Critique Manquant backend + frontend Suppression/édition occurrence individuelle d'une série récurrente impossible ▶
La spec US-823 montre un dialog 3 choix lorsque l'utilisateur tente de modifier ou supprimer une occurrence d'une indisponibilité récurrente : "Cet événement uniquement", "Cet événement et les suivants", "Tous les événements" — pattern identique à Google Calendar. Le backend actuel n'a aucun concept d'occurrence individuelle. Le DELETE et PATCH /api/indisponibilites/:id agissent toujours sur la série entière (seule ligne en base).
// Spec : dialog 3 choix → 3 actions backend distinctes nécessaires // 1. "Cet événement uniquement" // Manque : champ exceptions[] dans indisponibilite (dates à exclure) // 2. "Cet événement et les suivants" // Manque : PATCH recurrence_config.until = date_occurrence - 1 jour // 3. "Tous les événements" → DELETE existant (seul cas implémenté)
Spec US-823Modifier une seule occurrence → ajouter la date dans recurrence_config.exdates[]. Modifier "ce et suivants" → mettre à jour recurrence_config.until et créer une nouvelle série depuis cette date.
US-597 Majeure Contradiction backend Filtrage calendrier par technicien absent du endpoint missions/calendar ▶
Le composant mission-calendar.tsx affiche un filtre "Technicien" dans sa barre d'outils (filtre technicien_id), mais le endpoint GET /api/missions ne supporte ce filtre que pour les vues liste. La vue calendrier utilise le même endpoint mais ne transmet pas ce paramètre cohéremment — la logique de filtrage technicien existe dans certains chemins mais pas dans la query calendrier.
// mission-calendar.tsx — filtre technicien visible mais... // Vérifier si useMissions() dans la vue calendrier passe bien technicien_id GET /api/missions?view=calendar&technicien_id=uuid -- Le paramètre est accepté (ligne 79 de missions.ts) -- Mais la vue calendrier ne le transmet pas toujours correctement
US-813 Majeure Manquant backend Vue carte : géocodage des adresses non automatisé ▶
La spec US-813 mentionne un "géocodage des adresses via API Mapbox" déclenché automatiquement à la création ou modification d'une adresse de bâtiment. Les champs latitude / longitude existent dans adresse_batiment mais doivent être remplis manuellement. La vue carte (mission-map.tsx) ne peut afficher que les bâtiments qui ont des coordonnées — les autres sont invisibles.
// server/routes/batiments.ts — POST /:id/adresses (ligne ~270) INSERT INTO adresse_batiment (batiment_id, rue, code_postal, ville, latitude, longitude, ...) VALUES ($1, $2, $3, $4, $5, $6, ...) -- latitude et longitude viennent du body (requis manuellement) -- Manque : appel geocoding si lat/lng absents -- Ex: await mapboxGeocode(`${rue} ${code_postal} ${ville}`)
ImpactTous les bâtiments sans coordonnées manuelles sont invisibles sur la carte missions. Si le géocodage n'est pas implémenté, documenter que latitude/longitude sont requis à la saisie.
US-594 Majeure Contradiction frontend Picker tiers dans création mission ne filtre pas par rôle "locataire" ▶
La spec précise que le champ locataires dans create-mission-modal.tsx doit proposer un picker filtré sur les tiers ayant déjà un rôle de locataire (nb_edl_locataire > 0). Actuellement, la recherche via useTiersSearch retourne tous les tiers sans filtre de rôle. L'utilisateur voit propriétaires, mandataires et personnes morales dans la liste, ce qui est contre-intuitif et contraire à la spec.
// create-mission-modal.tsx — recherche locataires const { data: tiersList } = useTiersSearch(locSearch) // Devrait être : const { data: tiersList } = useTiersSearch(locSearch, { role: 'locataire' }) // Et côté backend : GET /api/tiers?search=X&role=locataire (déjà supporté !)
NoteLe backend supporte déjà ?role=locataire (ligne 41 de tiers.ts). C'est uniquement le frontend qui ne passe pas ce paramètre.
US-811 Majeure Manquant frontend Bandeau visuel "Mission terminée — édition limitée" absent ▶
La spec US-811 montre un bandeau d'alerte visible en haut de la fiche mission quand statut='terminee'. Ce bandeau informe l'utilisateur des champs encore modifiables (commentaire, dépôt des clés). Le code dans mission-detail-page.tsx implémente bien la logique isLocked qui désactive les champs, mais n'affiche aucun bandeau visuel explicatif. L'utilisateur ne comprend pas pourquoi les champs sont grisés.
// mission-detail-page.tsx — lignes 91-93 const isTerminated = mission?.statut === 'terminee' const isLocked = isTerminated || isCancelled // isLocked grise les champs correctement via prop locked={isLocked} // Manque : bandeau visible ↓ // { isTerminated && ( // <Alert variant="warning"> // Mission terminée — Édition limitée au commentaire et statut des clés // </Alert> // ) }
US-822 Mineure Manquant backend CleMission : photos du trousseau non supportées (table Photo entity_type='cle') ▶
La spec US-822 prévoit la possibilité d'attacher des photos à une clé (entity_type = 'cle' dans la table photo). La table photo dans le schéma supporte cet entity_type mais aucune route backend ni UI frontend ne permet de créer ou afficher ces photos. Dans la fiche mission, la section clés n'a pas de bouton d'upload photo.
// schema.sql — table photo entity_type VARCHAR(50) CHECK (entity_type IN ( 'edl', 'piece', 'item', 'cle' -- 'cle' présent mais jamais utilisé )) -- Manque : POST /api/edl/:id/cles/:cleId/photos
US-812 Mineure Incohérence frontend Kanban : statut "Actions en attente" sans couleur distincte de "Planifiée" ▶
La spec US-812 différencie visuellement les missions "Planifiée" (bleu) et "Planifiée avec actions en attente" (orange) par une bordure gauche colorée. Le Kanban montre un badge orange sur la carte (has_pending_actions), ce qui est correct, mais toutes les colonnes "planifiée" ont la même couleur de header. Une mission en attente de confirmation RDV et une mission sans action ont le même look visuel de colonne.
// mission-kanban.tsx // Le badge orange est présent (ligne 168-173) ✓ // Mais : couleur de la carte / bordure gauche ne change pas selon hasPending // Spec attendait : carte border-left orange si hasPending, bleu sinon
