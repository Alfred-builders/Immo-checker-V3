Securite & Integrite

45 problemes identifies — 5 critiques, 15 hauts

5 Problemes Critiques URGENT

1. Switch workspace sans authentification

POST /switch-workspace accepte userId du body sans verifier le token JWT. Un attaquant peut usurper n'importe quel user.

Fichier: server/routes/auth.ts:60-78 — Fix: utiliser uniquement req.user!.userId du token verifie

2. Tokens de reset password non haches

Les tokens UUID sont stockes en clair en DB. Si la base fuite, tous les resets en cours sont compromis.

Fix: hasher le token (bcrypt/sha256) avant stockage, comparer le hash

3. Race condition sur auto-terminaison EDL

Deux EDL signes en parallele peuvent declencher deux auto-terminaisons sans transaction.

Fix: wraper dans une transaction DB avec SELECT FOR UPDATE

4. Reference mission non-unique en DB

Aucune contrainte UNIQUE(workspace_id, reference). Deux creations simultanees = meme reference.

Fix: ajouter UNIQUE constraint ou utiliser une sequence DB

5. Refresh token replay possible

Le token est supprime apres usage mais deux requetes paralleles peuvent reutiliser le meme token.

Fix: rotation de refresh tokens avec invalidation atomique

15 Problemes Hauts

• Archive tiers/lots/batiments sans verification workspace complete

• Transition EDL non validee (signe → brouillon possible)

• Adresse update sans requireRole (tout user authentifie peut modifier)

• Race condition sur l'archivage (check + update non atomiques)

• Invitation disclosure possible via enumeration de tokens

• TiersOrganisation peut lier physique ↔ physique (pas de type check)

• Cascade delete manquant sur mission.lot_id

• Mission stats query inefficiente (full table scans multiples)

• Cle statut transition sans validation (deposee → remise possible)

• Index manquants sur mission(workspace_id, statut), edl_locataire(tiers_id)

• Archive flag non filtre sur certains detail endpoints

• Pas de rate limiting sur login/forgot-password

• Email lowercasing inconsistent entre endpoints

• Pagination cursor format inconsistent entre routes

• Dashboard stats pas invalides apres mutations mission

Analyse des Flows — Incoherences detectees

Problemes de logique metier et UX flows

Flow Login → Dashboard (incohérent)

Apres login, l'user est redirige vers /app/patrimoine (ancien defaut). Maintenant le dashboard existe, le redirect devrait etre /app/dashboard.

Statut : Corrige dans App.tsx mais login-page.tsx redirige encore vers /app/patrimoine

Fix : changer navigate('/app/patrimoine') → navigate('/app/dashboard') dans login-page.tsx et workspace-select-page.tsx

Flow Creation Tiers → Lien Lot (casse)

L'user cree un tiers depuis la page lot, ferme la modal, essaie de le lier comme proprietaire. Le search cache n'est pas invalide → le tiers n'apparait pas. Il faut rafraichir la page.

Fix : invalider queryKey ['search-tiers'] apres creation tiers

Flow Changement Role (dangereux)

Dans les parametres, changer le role d'un user (admin → gestionnaire) s'applique immediatement sans confirmation. Si l'admin se retire son propre role admin, il est lock-out.

Fix : ajouter dialog de confirmation + bloquer si c'est le dernier admin

Flow Colocation (confusing)

Quand on ajoute 2+ locataires, le radio "Type de bail" apparait mais rien n'indique ce qui se passe concretement (combien d'EDL seront crees, qui signe quoi). L'user doit deviner.

Suggestion : ajouter un bloc explicatif visuel "Bail individuel = 2 EDL separes" / "Bail collectif = 1 EDL signe par tous"

Flow Annulation Mission (pas clair)

Le bouton "Annuler" est sur la page detail et le drawer, mais pas sur la liste/kanban. L'user doit ouvrir chaque mission pour l'annuler. Pas de bulk action.

Suggestion : ajouter annulation depuis le kanban (right-click ou menu contextuel)

Flow Technicien refuse → rien ne se passe

Quand un technicien refuse une mission, la mission reste assignee avec statut_invitation="refuse" mais l'admin ne recoit aucune notification. Il doit verifier manuellement le bloc "Actions en attente".

Suggestion : notification push/email a l'admin quand invitation refusee
