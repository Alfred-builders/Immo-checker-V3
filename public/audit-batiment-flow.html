TL;DR

« Créer un immeuble c'est chiant. »

Le problème n'est pas le modèle de données — la hiérarchie Bâtiment → Lot reste correcte. Ce qu'il faut repenser c'est le parcours de création, principalement à l'intérieur de la modale de création de mission. L'utilisateur doit penser adresse avant de penser « bâtiment + lot ».

Recommandation principale
Proposition A — Address-first picker

Saisie de l'adresse en premier, on remonte les bâtiments existants à cette adresse. Couvre 90% des cas (un seul bâtiment) et règle l'edge case (résidence A/B/C).

Quick wins (faisables tout de suite)
  • • Désignation bâtiment → optionnelle
  • • Désignation lot → optionnelle
  • • Champ num_batiment exposé en UI
À explorer (plus loin)
Proposition C — Modale unifiée

Mission + Lot + Bâtiment dans une seule modale progressive. À challenger sur la complexité conditionnelle.

Retours client

Ce que dit Flat Checker

Friction « Créer un immeuble c'est chiant »
  • • Le workflow actuel demande de créer le bâtiment avant le lot — contre-intuitif.
  • • Sur les grosses résidences (Les Lilas A, B, C, D…), risque de doublons (« A maj / a min »).
  • • La désignation « Les Lilas » n'est pas systématique → ne devrait pas être obligatoire.
Position Alfred UX, pas modèle de données
  • • Garder Bâtiment → Lot en BDD.
  • • À la création d'un lot via l'adresse : remonter les bâtiments déjà présents.
  • • Désignation (« Les Lilas ») et numéro (A, B, C…) = deux concepts distincts.
Décisions à acter
  • ▸ Repenser le parcours de création lot dans la modale mission
  • ▸ Edge case « plusieurs bâtiments à la même adresse » : sélecteur
  • ▸ Désignation bâtiment → secondaire / non obligatoire
  • ▸ Champ « numéro de bâtiment » structuré (distinct désignation)
  • ▸ Désignation lot → également non obligatoire
  • ▸ À la création d'un lot : remonter bâtiments à l'adresse

Bonne nouvelle

Le champ num_batiment existe déjà dans la table batiment (schema.sql:136). Il n'est juste pas exposé en UI. Côté géocodage, l'app possède déjà un AddressAutocomplete (Mapbox FR) — il est juste utilisé en interne du formulaire bâtiment, pas en porte d'entrée.

Ce qui manque vraiment : (1) un endpoint GET /batiments?address=... ou un filtre ?lat,lng pour remonter les bâtiments à une adresse, (2) rendre designation nullable côté batiment et lot, (3) un sélecteur structuré pour num_batiment.

Proposition A · recommandée

Address-first picker dans la modale mission

L'utilisateur tape une adresse. L'app remonte les bâtiments existants à cette adresse (un ou plusieurs). Il choisit l'existant, sélectionne un lot, ou crée à la volée. Aucun changement de modèle de données — juste l'inversion du parcours.

Critères d'acceptation +
  • • La modale mission ouvre sur un champ adresse en focus, pas sur un champ lot.
  • • Une fois l'adresse géocodée, l'API retourne tous les bâtiments dans un rayon de 25 m (lat/lng) OU exact match rue + cp + ville.
  • • Si 0 bâtiment trouvé → CTA primaire « Créer un nouveau bâtiment ici » avec adresse pré-remplie.
  • • Si 1 bâtiment trouvé → présélectionné + bouton « ce n'est pas celui-là, créer un nouveau ».
  • • Si 2+ bâtiments trouvés → liste avec num_batiment mis en avant, désignation en secondaire.
  • • La création d'un bâtiment depuis cette modale ne demande que : type + num_batiment (optionnel) + designation (optionnel).

Proposition C · piste alternative

Modale unifiée Mission + Lot + Bâtiment

Une seule modale progressive qui adapte ses étapes selon le contexte. Plus puissante mais plus complexe à concevoir et à tester. À garder pour V1.1 si la Proposition A ne suffit pas.

Questions ouvertes

À trancher avant dev

Bâtiments non lettrés

Si la résidence n'utilise ni lettre ni numéro, faut-il afficher « Bâtiment principal » par défaut ou laisser vide ? Hypothèse : si num_batiment est vide ET désignation vide → fallback « Bâtiment unique » dans la liste.

Maison individuelle

CLAUDE.md dit « 1 Bâtiment + 1 Lot ». L'address-first picker doit auto-skip la sélection bâtiment si type = maison.

Fallback affichage du lot sans désignation

Si lot.designation est vide, afficher {type_bien} · étage {etage} · {emplacement_palier}. À valider avec le client.

Tolérance géographique

Match strict rue + cp + ville ou rayon GPS de 25 m ? Le strict est plus simple côté UX (pas d'erreurs d'arrondi) mais risque de rater une saisie « 12 r. des Lilas » vs « 12 rue des Lilas ». Recommandation : strict sur l'adresse normalisée Mapbox.
