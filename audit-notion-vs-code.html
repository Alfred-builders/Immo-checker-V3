Contradictions detectees entre differentes User Stories, EPICs, ou entre specs et decisions CLAUDE.md.

Critique Auto-creation EDL : US-594 vs US-811
US-594 dit que les EDL sont auto-crees a la creation de mission selon le sens + inventaire + type de bail.
US-811 mentionne un bouton "Ajouter EDL" sur la page detail, ce qui implique un ajout a la demande.

Question : Les EDL sont-ils crees automatiquement OU ajoutes manuellement apres coup ? Les deux ne sont pas contradictoires si la creation auto couvre le cas initial et le bouton permet d'en ajouter supplementaires — mais ce n'est pas explicite dans les specs.

Etat code : Les EDL sont crees automatiquement a la creation de mission (backend missions.ts POST). Le bouton "Ajouter EDL" n'existe pas dans le detail page.
EPIC 13 US-594 US-811 EDL lifecycle
Critique Locataire optionnel a la creation : vide = placeholder ou erreur ?
US-594 accepte un champ locataire vide : "Mission created + 1 EDL without tenant (add later via US-811)".
Mais EPIC 13 definit le workflow EDL comme "entrant/sortant" necessitant des locataires pour la signature.

Probleme : Un EDL sans locataire n'a pas de signataire. Est-ce un brouillon non-signable, ou un placeholder pour ajout ulterieur ? L'auto-terminaison attend "tous EDL signes" — un EDL sans locataire bloque-t-il la terminaison indefiniment ?

Etat code : Le code permet la creation sans locataire. L'EDL est en brouillon. Le locataire peut etre ajoute plus tard. Pas de logique de blocage explicite.
EPIC 13 US-594 Data integrity
Haut Revalidation UX : page detail vs drawer dashboard
US-811 dit que la modification date/heure sur la page detail declenche une modale de revalidation avec choix (demander revalidation OU confirmer tel quel).
US-842 dit la meme chose pour le drawer dashboard.

Etat code : La page detail utilise un RevalidationDialog (modale). Le drawer utilise un pattern different (inline ou simplifie). Les deux devraient etre identiques selon les specs.
EPIC 13 EPIC 14 US-811 US-842 UX consistency
Haut Mission.statut = "assignee" : quand exactement ?
CLAUDE.md 7d dit : "planifiee → assignee : quand technicien accepte l'invitation".
US-594 dit : le statut initial est "assignee" si un technicien est selectionne a la creation (avant meme que le tech n'accepte).

Contradiction : Un tech selectionne mais qui n'a pas encore accepte = planifiee ou assignee ? Les deux logiques coexistent dans le code. La creation met "assignee" si tech present, mais les transitions d'etat disent que "assignee" = tech a accepte.

Les cles a deposer sont dans le drawer (US-842) mais pas dans le compteur "Actions en attente".

Points dans les specs qui semblent illogiques, sur-ingenies, ou qui meritent une remise en question.

Logique CleMission FK vers edl_id au lieu de mission_id
La justification est "colocation geree" — chaque co-locataire a ses propres cles.

Probleme : Dans la grande majorite des cas (95%+), il y a 1 seul EDL par mission. La FK vers edl_id ajoute de la complexite (jointure supplementaire pour afficher les cles d'une mission) sans benefice tangible pour le cas simple. De plus, les cles physiques sont les memes (une serrure = un jeu de cles), c'est le document de remise qui differe par co-locataire, pas les cles elles-memes.

Suggestion : FK vers mission_id + champ optionnel edl_id pour les cas de colocation specifique. Simplifie 95% des queries.
Architecture Over-engineering CleMission
Logique 3 statuts separes sur Mission (statut + statut_rdv + statut_invitation)
Chaque mission a 3 axes de statut independants :
- statut : planifiee | assignee | terminee | annulee
- statut_rdv : a_confirmer | confirme | reporte
- MissionTechnicien.statut_invitation : en_attente | accepte | refuse

Probleme : Cela cree 4 × 3 × 3 = 36 combinaisons d'etats possibles. Beaucoup sont invalides (ex: terminee + refuse). Aucune matrice de validite n'est definie dans les specs. Le "Actions en attente" du dashboard filtre sur ces 3 axes simultanement, rendant la logique fragile.

Suggestion : Definir explicitement les combinaisons valides et ajouter des contraintes DB (CHECK ou trigger) pour empecher les etats incoherents.
State machine Complexite Validation
Logique Record Picker avec creation en cascade (3 niveaux de modales)
US-594 decrit un flow de creation mission ou :
1. Modale creation mission
2. → Picker lot → bouton "Creer lot" → sous-modale lot
3. → Picker batiment → bouton "Creer batiment" → sous-sous-modale batiment

Probleme : 3 niveaux de modales empilees = UX lourde et disorienting. Chaque sous-modale doit gerer son propre etat + auto-selectionner le resultat au retour.

Etat code : Le code implemente 2 niveaux (mission → lot creation). Le 3eme niveau (batiment creation dans lot creation) est present dans create-lot-modal.tsx qui peut appeler create-building-modal.tsx.

Suggestion : Limiter a 2 niveaux max. Si le batiment n'existe pas, rediriger vers la page patrimoine plutot qu'ouvrir une 3eme modale.
UX Over-engineering US-594
Logique Absence de gestion locative MAIS affichage "dernier locataire"
CLAUDE.md 7b : "PAS de gestion locative — l'occupation des logements n'est pas geree".
Pourtant, EPIC 1 affiche "dernier locataire" sur la fiche lot, et EPIC 2 montre "lots passes" du locataire.

Paradoxe : Sans gestion locative, le "dernier locataire" est derive des EDL les plus recents. Mais si un locataire change sans qu'il y ait d'EDL (rare mais possible), l'info est fausse. L'affichage cree une attente de precision que le systeme ne peut pas garantir.

Suggestion : Soit ajouter un label "Dernier locataire (selon EDL)" pour gerer l'attente, soit ne pas l'afficher du tout.
